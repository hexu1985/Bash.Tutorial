`sar` 监控磁盘 I/O 有高频和低频两个视角，分别适用于排查实时瓶颈和分析总体吞吐量。

### 🎯 核心理念：`-b` vs `-d`
监控磁盘 I/O 最核心的两个选项是 `-b` 和 `-d`，它们的侧重点很不一样，简单来说：`-b` 是“宏观统计看大盘”，适合快速判断系统整体 I/O 压力；`-d` 是“微观定位到盘”，适合定位具体哪个磁盘出了问题。了解了这两者的区别，具体的用法就一目了然了。

### 📊 高频视角 (排查瓶颈)：`sar -b` (I/O 与传输速率)
`-b` 报告的是系统全局的 I/O 统计摘要，非常实用，一般用它来快速判断或持续监控系统的整体 I/O 压力。

*   **命令示例**：`sar -b 1 5` (每1秒采样1次，共5次)
*   **典型输出**：

```text
Linux 3.10.0-957.el7.x86_64 (hostname)     04/28/2026      _x86_64_        (4 CPU)

01:27:32 PM       tps      rtps      wtps   bread/s   bwrtn/s
01:27:33 PM     120.00     10.00    110.00    560.00   4480.00
01:27:34 PM     145.00     15.00    130.00    840.00   5200.00
01:27:35 PM     110.00      8.00    102.00    448.00   4080.00
Average:        125.00     11.00    114.00    616.00   4586.67
```

*   **字段含义**：
    *   **`tps`**：**每秒传输的 I/O 请求总数**。`rtps` + `wtps` = `tps`。
    *   **`rtps` / `wtps`**：分别对应每秒的读、写请求总数。
    *   **`bread/s` / `bwrtn/s`**：每秒读写的数据量，单位是“块/秒”。在 Linux 中，一个块通常等于 512 字节。

### 🚨 微观视角 (定位问题盘)：`sar -d` (每个块设备)
`-d` 能细致到每个块设备逐一报告，字段也丰富得多。配合 `-p` 参数，输出中的设备名会更清晰（如 `sda`）。

*   **命令示例**：`sar -d -p 1 3` (每1秒采样1次，共3次)
*   **典型输出**：

```text
Linux 3.10.0-957.el7.x86_64 (hostname)     04/28/2026      _x86_64_        (4 CPU)

01:27:32 PM       DEV       tps     rkB/s     wkB/s   areq-sz   aqu-sz     await     svctm     %util
01:27:33 PM       sda    120.00    560.00   4480.00    42.00      2.50     20.85      8.33    100.00
01:27:34 PM       sda    145.00    840.00   5200.00    41.66      3.20     22.07      6.90    100.00
01:27:35 PM       sda    110.00    448.00   4080.00    41.16      1.80     16.36      9.09    100.00
Average:          sda    125.00    616.00   4586.67    41.62      2.50     20.00      8.00    100.00
```

*   **字段含义**：
    *   **`DEV`**：设备名（例如 `sda`、`sdb`）。
    *   **`tps`**：同 `-b` 选项。
    *   **`rkB/s` / `wkB/s`**：每秒读写的数据量，单位是 **KB**。
    *   **`areq-sz`**：**平均每次 I/O 请求的数据量 (KB)**，值越大，说明请求的连续性越好。
    *   **`aqu-sz`**：**平均 I/O 请求队列长度**，持续大于1，表示有大量 I/O 在排队。
    *   **`await`**：**平均每次 I/O 请求的响应时间 (毫秒)**。
    *   **`svctm`**：**平均每次 I/O 请求的服务时间 (毫秒)**。（**⚠️ 注意：从 `sysstat` 12.x 版本起此字段已废弃，可忽略。**）
    *   **`%util`**：**设备繁忙度 (%)**。

> 如果 `await` 远大于 `svctm`（或平均服务时间），说明是系统瓶颈导致请求排队，而非磁盘本身慢。

### 💾 保存与读取磁盘 I/O 数据
`sar` 强大之处在于能定期采样系统活动并保存，为事后分析提供依据。

#### 1. 手动保存数据 (`-o`)
使用 `-o` 选项可将实时监控结果以二进制形式存入文件，便于事后分析。
*   **命令**：`sar -d -p -o /tmp/disk_io_sar.bin 2 5` (每2秒采样1次，共5次，结果保存到 `/tmp/disk_io_sar.bin`)
*   **特点**：该文件为特殊格式，**请勿用 `cat` 或 `vim` 直接查看，会显示乱码**。

#### 2. 从文件读取数据 (`-f`)
使用 `-f` 选项可解析并读取之前保存的二进制文件，重现完整报告。
*   **命令**：`sar -d -p -f /tmp/disk_io_sar.bin`
*   **高级查询**：结合 `-s` 和 `-e` 选项可精准查询特定时间段。
    *   **命令**：`sar -d -p -f /var/log/sa/sa28 -s 14:00:00 -e 15:00:00` (查询28号14:00至15:00期间的I/O报告)

### ⚠️ 重要注意事项
*   **瓶颈识别**：`%util` 持续接近 100% 意味着磁盘已饱和，是 I/O 瓶颈的明显信号。此时 `await` 也会相应升高。
*   **基准环境**：`await` 的理想值取决于物理硬件，**固态硬盘 (SSD) 远低于机械硬盘 (HDD)**。应在系统正常时建立性能基准线以供对比。
*   **系统开销**：`sar` 本身开销很小，但过短的采样间隔（<1秒）在大规模生产环境中仍可能产生轻微负载。
*   **数据精度**：`sar` 依赖内核计数器，在处理大量短时 I/O 时数据存在理论上的精度限制。
*   **IOPS vs 吞吐量**：I/O 性能需综合考量 **IOPS**（对应 `tps`）和 **吞吐量**（对应 `rkB/s`、`wkB/s`）。大量小文件读写看 IOPS，大文件读写则看吞吐量。

### 💎 总结
监控磁盘 I/O 时，先通过 `sar -b` 查看系统整体负载，确认存在 I/O 压力后，再用 `sar -d -p` 定位到具体的繁忙磁盘，分析 `%util` 和 `await` 等关键指标来评估性能。通过 `-o` 保存数据并用 `-f` 读取，可以实现深度的离线分析，是排查历史问题的标准方法。
