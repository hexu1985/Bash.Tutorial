## 一、sar命令概述

`sar` 是 `System Activity Reporter` 的缩写，是 Linux 系统中最全面的性能监控工具之一。它能够收集并报告系统的各项核心运行数据，包括 CPU 使用率、内存用量、磁盘 I/O 活动、网络流量、内核状态等。

`sar` 的核心价值在于：
- **实时监控**：按指定间隔实时采样并输出当前系统状态
- **历史回溯**：系统后台持续收集并保存数据，支持事后分析任意时间段的性能走势
- **全面覆盖**：涵盖 CPU、内存、磁盘、网络、内核等几乎所有核心子系统

`sar` 命令属于 `sysstat` 软件包，可通过 `yum install sysstat` 或 `apt-get install sysstat` 安装。

## 二、语法与核心参数

### 2.1 基本语法

```
sar [options] [interval] [count]
```

| 组成部分 | 说明 |
|---------|------|
| `interval` | 采样间隔（秒），必须手动设置 |
| `count`   | 采样次数，默认为 1；若只指定 interval 不指定 count，则连续不间断采样 |
| `options` | 指定要监控哪类资源 |

### 2.2 常用选项速查

| 选项 | 用途 |
|------|------|
| `-u` | 显示 CPU 使用情况（%user、%system、%idle等） |
| `-r` | 显示内存和交换空间使用情况 |
| `-b` | 显示磁盘 I/O 和传输速率统计 |
| `-d` | 显示每个块设备的使用状态 |
| `-q` | 显示系统平均负载和运行队列长度 |
| `-n DEV` | 显示网络设备接收/发送的数据包和字节数 |
| `-n SOCK` | 显示 socket 连接信息 |
| `-P ALL` | 分别显示每个 CPU 核心的使用情况 |
| `-w` | 显示系统交换活动状态 |
| `-v` | 显示 inode、文件和其他内核表的状态 |
| `-A` | 显示所有报告信息 |
| `-o file` | 将结果以二进制格式保存到文件 |
| `-f file` | 从指定文件中读取报告（用于查看历史数据） |

## 三、各场景性能指标详细解读

### 3.1 CPU 使用情况（`sar -u`）

```
$ sar -u 1 3
Linux 3.10.0-957.el7.x86_64 (hostname)     04/24/2026      _x86_64_        (4 CPU)

06:18:23 AM     CPU     %user     %nice   %system   %iowait    %steal     %idle
06:18:24 AM     all     12.11      0.00      2.77      3.11      0.00     82.01
06:18:25 AM     all      6.55      0.00      2.07      0.00      0.00     91.38
Average:        all      9.33      0.00      2.42      1.55      0.00     86.70
```

| 指标 | 含义 | 健康阈值参考 |
|------|------|------------|
| `%user` | 用户态进程消耗的 CPU 百分比 | 持续 >70% 需关注应用性能 |
| `%nice` | 低优先级用户进程（nice值调整）消耗的 CPU 百分比 | 通常较低 |
| `%system` | 内核态系统调用消耗的 CPU 百分比 | 持续 >30% 可能存在系统调用瓶颈 |
| `%iowait` | CPU 空闲且等待磁盘 I/O 的时间百分比 | 持续 >20% 说明磁盘是性能瓶颈 |
| `%steal` | 虚拟化环境中被 hypervisor 抢占的时间百分比 | 持续 >10% 说明宿主机负载过高 |
| `%idle` | CPU 完全空闲的时间百分比 | 持续 <10% 说明系统接近满载 |

### 3.2 内存使用情况（`sar -r`）

```
$ sar -r 1 2
Linux 3.10.0-957.el7.x86_64 (hostname)     04/24/2026      _x86_64_        (4 CPU)

06:18:23 AM kbmemfree kbmemused  %memused kbbuffers kbcached  kbcommit   %commit
06:18:24 AM    245678   1543222     86.25     45678    345678   1250000     58.20
```

| 指标 | 含义 |
|------|------|
| `kbmemfree` | 空闲物理内存（KB），不含 buffer/cache |
| `kbmemused` | 已用物理内存（KB），包含 buffer/cache |
| `%memused` | 已用内存占总内存的百分比 |
| `kbbuffers` | 内核缓冲区占用的内存（KB） |
| `kbcached` | 页面缓存占用的内存（KB） |
| `kbcommit` | 系统承诺分配的内存总量（KB），含 swap |
| `%commit` | 承诺内存占总内存（含 swap）的百分比 |

### 3.3 磁盘 I/O 情况（`sar -b` 或 `sar -d`）

**`sar -b` — I/O 传输速率统计：**

| 指标 | 含义 |
|------|------|
| `tps` | 每秒向设备发出的 I/O 请求总数 |
| `rtps` | 每秒读请求次数 |
| `wtps` | 每秒写请求次数 |
| `bread/s` | 每秒读数据量（块/秒） |
| `bwrtn/s` | 每秒写数据量（块/秒） |

**`sar -d` — 每个块设备详情：**

| 指标 | 含义 | 健康阈值参考 |
|------|------|------------|
| `await` | 平均每次 I/O 请求的等待时间（毫秒） | < 10ms 正常，> 50ms 说明磁盘响应慢 |
| `svctm` | 平均每次 I/O 服务时间（毫秒） | 与 await 接近说明磁盘本身无积压 |
| `%util` | 设备忙处理 I/O 的时间百分比 | 持续 > 80% 说明磁盘已达瓶颈 |

### 3.4 网络活动（`sar -n DEV / SOCK`）

**DEV 类 — 网络设备接口统计：**

| 指标 | 含义 | 定位方向 |
|------|------|---------|
| `IFACE` | 网络设备名称（如 eth0、ens33） | 定位具体网卡 |
| `rxpck/s` | 每秒接收的数据包数 | 流量突增定位 |
| `txpck/s` | 每秒发送的数据包数 | 流量突增定位 |
| `rxkB/s` | 每秒接收的字节数（KB） | 带宽瓶颈分析 |
| `txkB/s` | 每秒发送的字节数（KB） | 带宽瓶颈分析 |
| `rxmcst/s` | 每秒接收的多播包数 | 多播风暴排查 |

**SOCK 类 — Socket 连接信息：**

| 指标 | 含义 |
|------|------|
| `totsck` | 系统已使用的 socket 总数 |
| `tcpsck` | 当前 TCP 协议使用的 socket 数量 |
| `udpsck` | 当前 UDP 协议使用的 socket 数量 |
| `rawsck` | 当前 RAW 协议使用的 socket 数量 |

**EDEV 类 — 网络设备错误统计：**

该类别专门用于排查网络通信失败和错误，包括收发包错误数、碰撞数、丢帧数等。

### 3.5 系统负载与进程（`sar -q`）

| 指标 | 含义 | 健康阈值参考 |
|------|------|------------|
| `ldavg-1/5/15` | 过去 1/5/15 分钟的平均负载 | 不宜持续超过 CPU 核心数 |
| `runq-sz` | 运行队列中等待执行的进程数 | 持续高于 CPU 核心数说明 CPU 不足 |
| `plist-sz` | 系统中的进程总数 | 异常突增需排查僵尸进程 |

## 四、配置与数据管理

### 4.1 数据收集机制

`sar` 通过 cron（或 systemd timer）定期调用 `sa1` 脚本采集数据并写入二进制数据文件，再由 `sa2` 脚本生成每日汇总报告。

| 文件位置 | 文件类型 | 说明 |
|---------|---------|------|
| `/var/log/sa/saDD` | 二进制 | 当日活动数据（DD 为日期） |
| `/var/log/sa/sarDD` | 文本 | 每日汇总报告，由 `sa2 -A` 生成 |
| `/etc/cron.d/sysstat` | 配置 | 定义采集频率（默认每 10 分钟） |
| `/etc/default/sysstat` | 配置 | 定义数据保留天数等行为 |

### 4.2 查看历史数据的典型方法

```bash
# 直接查看当日已收集的所有数据
sar

# 查看昨天的数据
sar -f /var/log/sa/sa$(date +%d -d "yesterday")

# 查看指定时间段的数据（CPU 维度，8:00-10:00）
sar -u -s 08:00:00 -e 10:00:00

# 查看特定日期的数据（如当月 15 号）
sar -f /var/log/sa/sa15 -u
```

- **`-f filename`**：从指定二进制文件中读取历史记录
- **`-s hh:mm:ss`**：设定报告的起始时间
- **`-e hh:mm:ss`**：设定报告的结束时间
- **`-1 / -2 ...`**：直接查看昨天/前天的数据，`-1` 表示昨日的系统活动文件

### 4.3 常见配置修改

**启用历史数据自动收集**：编辑 `/etc/default/sysstat`，将 `ENABLED="false"` 改为 `"true"`

**调整保留天数**：编辑 `/etc/sysconfig/sysstat`，修改 `HISTORY=7`（默认 60 天）

**调整采集频率（提高精度）**：编辑 `/etc/cron.d/sysstat`，将 `*/10 * * * *` 改为 `*/2 * * * *`（每 2 分钟采集一次）

## 五、常用实战场景示例

| 场景 | 命令示例 | 输出内容 |
|------|---------|---------|
| **实时监控 CPU** | `sar -u 1 5` | 每秒采一次，共 5 次，输出 CPU 各指标 |
| **实时监控内存** | `sar -r 1 5` | 每秒采一次，共 5 次，输出内存用量 |
| **实时监控磁盘 I/O** | `sar -b 1 5` | 每秒采一次，共 5 次，输出 tps、读写速率 |
| **实时监控网络** | `sar -n DEV 1 5` | 每秒采一次，共 5 次，输出各网卡收发数据 |
| **查看每个 CPU 核心** | `sar -P ALL 2 3` | 每 2 秒采一次，共 3 次，分别输出每个核心的状态 |
| **监控 socket 连接** | `sar -n SOCK 1 1` | 输出 socket 总数、TCP/UDP 使用量 |
| **监控交换活动** | `sar -w 1 5` | 每秒采一次，输出页面换入/换出统计 |
| **保存实时数据** | `sar -u -o cpu_log 1 10` | 将 CPU 数据以二进制保存至 cpu_log |
| **读取历史数据** | `sar -u -f cpu_log` | 从 cpu_log 中读取并显示报告 |
| **查看持续负载** | `sar -q 5` | 每 5 秒输出一次平均负载和运行队列 |

## 六、总结

`sar` 作为系统运维的核心工具，其强大的历史回溯能力和全面的资源覆盖，使其成为性能问题排查中不可或缺的利器。日常建议开启 sysstat 后台数据收集服务（默认每 10 分钟采集一次），并配合 `sar -u/-r/-b/-n DEV` 等核心选项建立常规巡检基线。当遇到性能问题时，通过 `-s/-e` 结合历史数据精确定位峰值时段，再结合 `-P ALL`、`-d`、`-n EDEV` 等针对性选项深入分析，是高效解决问题的最佳实践路径。

> **提示**：不同 Linux 发行版支持的选项可能略有差异，`man sar` 可查阅完整的手册页。
