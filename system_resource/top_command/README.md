`top`是 Linux 下一个强大的实时系统监控工具，常被称为“性能分析神器”。它就像Linux系统的任务管理器，能动态、实时地展示系统当前的运行状态，包括CPU使用率、内存占用以及每个进程的详细信息。

---


## 二、`top` 输出示例及逐行解释

```
top - 15:23:45 up 10 days,  2:15,  3 users,  load average: 0.08, 0.03, 0.01
Tasks: 124 total,   1 running, 123 sleeping,   0 stopped,   0 zombie
%Cpu(s):  2.3 us,  0.5 sy,  0.0 ni, 97.0 id,  0.2 wa,  0.0 hi,  0.0 si,  0.0 st
MiB Mem :   7822.4 total,   1023.5 free,   2100.2 used,   4698.7 buff/cache
MiB Swap:   2048.0 total,   2048.0 free,      0.0 used.   5231.2 avail Mem

    PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
   1234 root      20   0  562123  98765  12345 S   5.0   1.2   10:23.45 python3 app.py
   5678 mysql     20   0 2350000 145678  23456 S   2.5   1.8  45:12.30 mysqld
   9101 nginx     20   0  124000  24567   8901 S   0.3   0.3   0:15.67 nginx
   ...
```

下面分区域解释。

### 1️⃣ 第一行：系统概况

```
top - 15:23:45 up 10 days,  2:15,  3 users,  load average: 0.08, 0.03, 0.01
```

| 字段 | 含义 |
|------|------|
| `15:23:45` | 当前系统时间 |
| `up 10 days, 2:15` | 系统已运行 10 天 2 小时 15 分钟 |
| `3 users` | 当前有 3 个用户登录（包括本地和远程） |
| `load average: 0.08, 0.03, 0.01` | 过去 1 分钟、5 分钟、15 分钟的平均负载。简单理解：数值低于 CPU 核心数则系统较空闲，此处 0.08 表示负载很低。 |

### 2️⃣ 第二行：进程统计

```
Tasks: 124 total,   1 running, 123 sleeping,   0 stopped,   0 zombie
```

| 字段 | 含义 |
|------|------|
| `124 total` | 总进程数 |
| `1 running` | 正在运行（或可运行）的进程数 |
| `123 sleeping` | 睡眠中的进程（等待某些事件） |
| `0 stopped` | 被暂停的进程（如通过 `Ctrl+Z`） |
| `0 zombie` | 僵尸进程（已结束但未被父进程回收）。若不为 0 通常表明有问题。 |

### 3️⃣ 第三行：CPU 状态

```
%Cpu(s):  2.3 us,  0.5 sy,  0.0 ni, 97.0 id,  0.2 wa,  0.0 hi,  0.0 si,  0.0 st
```

| 字段 | 含义 |
|------|------|
| `us` (user) | 用户态进程占用 CPU 百分比 |
| `sy` (system) | 内核态（系统调用、内核代码）占用 CPU 百分比 |
| `ni` (niced) | 改变过优先级（nice 值）的用户进程占用的 CPU |
| `id` (idle) | 空闲 CPU 百分比 |
| `wa` (iowait) | 等待磁盘 I/O 完成的 CPU 时间占比（若持续高，说明磁盘慢） |
| `hi` (hardware interrupts) | 处理硬件中断的 CPU 时间 |
| `si` (software interrupts) | 处理软件中断的 CPU 时间 |
| `st` (steal time) | 虚拟化环境中被宿主机“偷走”的 CPU 时间（虚拟机专用） |

**示例解读**：CPU 总体很空闲（97% idle），用户程序用了 2.3%，内核用了 0.5%，几乎没有 I/O 等待。

### 4️⃣ 第四、五行：内存与交换分区

```
MiB Mem :   7822.4 total,   1023.5 free,   2100.2 used,   4698.7 buff/cache
MiB Swap:   2048.0 total,   2048.0 free,      0.0 used.   5231.2 avail Mem
```

| 字段 | 含义 |
|------|------|
| `Mem total` | 物理内存总量 |
| `Mem free` | 完全未使用的物理内存 |
| `Mem used` | 已使用的物理内存（含缓存） |
| `buff/cache` | 页面缓存和缓冲区占用的内存（Linux 会将其用于加速文件读写） |
| `Swap total` | 交换分区总量 |
| `Swap free` | 空闲交换分区 |
| `Swap used` | 已用交换分区（若很大说明物理内存不足） |
| `avail Mem` | 可用于启动新程序的估计内存量（不需交换） |

**示例解读**：总内存 7.8GB，实际活跃使用约 2.1GB，还有 4.7GB 作为缓存，可随时释放给应用。交换分区完全没用上。

### 5️⃣ 进程列表区域

```
    PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
   1234 root      20   0  562123  98765  12345 S   5.0   1.2   10:23.45 python3 app.py
   5678 mysql     20   0 2350000 145678  23456 S   2.5   1.8   45:12.30 mysqld
```

| 字段 | 含义 |
|------|------|
| `PID` | 进程唯一标识号（用于 kill、renice 等操作） |
| `USER` | 启动该进程的用户名 |
| `PR` | 内核调度优先级（数字越小优先级越高） |
| `NI` (nice) | 用户可修改的优先级，范围 -20（最高）到 19（最低） |
| `VIRT` | 虚拟内存总量（进程“申请”的所有内存，包括未实际分配的） |
| `RES` (resident) | 实际占用的物理内存（**重点关注**，单位 KB） |
| `SHR` (shared) | 共享内存大小（与其他进程共享的库等） |
| `S` (status) | 进程状态：<br>- `R`：运行中或可运行<br>- `S`：可中断睡眠<br>- `D`：不可中断睡眠（常与磁盘 I/O 相关）<br>- `Z`：僵尸进程<br>- `T`：被停止 |
| `%CPU` | CPU 使用率（多核可能超过 100%） |
| `%MEM` | 物理内存占用率 = RES / total_memory |
| `TIME+` | 进程自启动以来累计使用的 CPU 时间（分钟:秒.百分秒） |
| `COMMAND` | 命令名或完整命令行（按 `c` 可切换） |

**注意**：`VIRT`、`RES`、`SHR` 默认单位是 KiB（千字节），但不同 `top` 版本可能用 `MiB` 显示，以实际为准。

## 三、常用交互命令（按键示例）

在 `top` 运行界面中，直接按以下键（无需回车）：

| 按键 | 作用 | 示例效果 |
|------|------|----------|
| `P` (大写) | 按 `%CPU` 降序排序 | 立刻看到最耗 CPU 的进程 |
| `M` (大写) | 按 `%MEM` 降序排序 | 找到内存占用最多的进程 |
| `1` (数字) | 展开/折叠多核 CPU | 按一下会变成：<br>`Cpu0: 5% us, ...`<br>`Cpu1: 2% us, ...`<br>再按恢复总和 |
| `c` | 切换显示完整命令行 | 将 `python3` 变成 `python3 /app/main.py --port 8080` |
| `k` | 杀死进程 | 先提示输入 PID，再输入信号（如 `9` 强制杀死） |
| `u` | 仅显示某用户的进程 | 输入 `mysql` 后，只看到 mysql 相关进程 |
| `s` | 修改刷新间隔（秒） | 输入 `1` 变成 1 秒刷新一次（注意：对系统负载有轻微影响） |
| `f` | 进入字段管理界面 | 可以添加或删除显示的列（如添加 `PPID` 父进程ID） |
| `o` | 动态过滤 | 例如输入 `COMMAND=nginx` 只显示 nginx 进程 |
| `q` | 退出 top | - |

## 四、命令行启动示例（非交互模式）

有时你需要把 `top` 的输出保存到文件或只运行一次，可以使用命令行参数：

### 1. 固定刷新间隔运行

```bash
top -d 2
```
每 2 秒刷新一次（默认 3 秒）。

### 2. 只显示特定用户进程

```bash
top -u mysql
```
仅监视 `mysql` 用户的进程。

### 3. 只显示指定 PID 的进程（可多个）

```bash
top -p 1234 -p 5678
```

### 4. 批处理模式 – 输出一次后退出（适合脚本）

```bash
top -b -n 1 > /tmp/top_snapshot.txt
```
- `-b`：批处理模式（不进入交互界面）
- `-n 1`：只采集一次，然后自动退出

生成的文件内容与交互式 top 相同，可以直接用文本编辑器查看。

### 5. 显示线程（用于分析多线程应用）

```bash
top -H -p 1234
```
只显示 PID 为 1234 的进程内部的所有线程，并可以看到每个线程的 CPU 占用。

## 五、结合实际场景的示例说明

### 场景 1：系统突然变慢，如何定位最耗 CPU 的进程？

1. 运行 `top`
2. 按 `P`（大写），让进程按 `%CPU` 降序排列
3. 观察最上面的进程，比如看到 `%CPU` 为 150% 的 `java` 进程
4. 按 `c` 查看其完整命令行，确认是哪个应用
5. 若需要终止它，按 `k`，输入该 PID，再输入信号 `15` (正常终止) 或 `9` (强制杀死)

### 场景 2：怀疑内存泄漏，如何查看内存占用最高的进程？

1. 运行 `top`
2. 按 `M`（大写），按 `%MEM` 排序
3. 观察 `RES` 列，看哪个进程占用物理内存异常大
4. 也可以按 `f` 添加 `RSAN` 等更多内存字段辅助分析

### 场景 3：只想看某个特定服务（如 nginx）的实时状态

- **方法一**：启动时过滤  
  ```bash
  top -p $(pgrep -d',' nginx)
  ```
- **方法二**：进入 `top` 后，按 `o`，输入 `COMMAND=nginx`

## 六、小技巧总结

| 你想做的事 | 怎么做 |
|-----------|--------|
| 找出 CPU 飙升的进程 | `top` → 按 `P` |
| 找出内存消耗大户 | `top` → 按 `M` |
| 看具体每个 CPU 核心负载 | `top` → 按 `1` |
| 看完整命令（含参数） | `top` → 按 `c` |
| 临时调整刷新间隔为 0.5 秒 | `top` → 按 `s` → 输入 `0.5` |
| 杀死一个“卡死”的进程 | `top` → 按 `k` → 输入 PID → 输入 `9` |
| 只监控自己的进程 | `top` → 按 `u` → 输入你的用户名 |
| 只运行一次并存入文件 | `top -b -n 1 > top.txt` |
